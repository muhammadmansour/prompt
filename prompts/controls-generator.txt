## Instructions

For EACH requirement above, generate 1–3 applied controls tailored to the organization's industry vertical, size, compliance maturity level, regulatory mandates, governance structure, data classification, and infrastructure type.

**Tailoring rules:**
- Low maturity orgs (1–2): simpler, foundational controls; fewer controls per requirement
- High maturity orgs (4–5): advanced, layered controls; consider automation and continuous monitoring
- Sector-specific: inject industry-specific practices (e.g. PCI DSS for banking, HIPAA for healthcare)
- Size-aware: enterprise orgs need formal governance; small orgs need pragmatic lightweight controls
- Data classification: Restricted/Confidential data demands stricter controls
- Cloud-native vs on-premise vs hybrid: adapt technical implementation guidance accordingly

**Deduplication rules (CRITICAL):**
- If TWO OR MORE requirements can be satisfied by the SAME control, generate that control ONCE and list ALL the requirement refIds it covers in the `for_requirements` array.
- Do NOT generate near-duplicate controls with slightly different names for different requirements. Merge them.
- A single control CAN and SHOULD cover multiple requirements when logically appropriate.

Each control MUST include ALL of these fields:
1. **for_requirements**: Array of requirement refIds this control covers (e.g. ["5.1.a", "5.1.b"]). MUST contain at least one refId.
2. **name**: A clear, concise control name (English, 5-15 words)
3. **name_ar**: Arabic translation of the control name
4. **description**: Detailed description of what the control entails, how to implement it, and what evidence demonstrates it (English, 30-80 words)
5. **description_ar**: Arabic translation of the description
6. **category**: One of: "policy", "process", "technical", "physical"
7. **csf_function**: One of: "govern", "identify", "protect", "detect", "respond", "recover"
8. **status**: Always "to_do"
9. **priority**: Integer 1–4 where 1 = Critical, 2 = High, 3 = Medium, 4 = Low
10. **effort**: One of: "S", "M", "L", "XL"
11. **relevance_score**: Integer 0–100 indicating how relevant this control is to the specific organization
12. **evidence_examples**: Array of 2-3 specific evidence items
13. **implementation_guidance**: Brief practical implementation guidance specific to the org (2-3 sentences)
